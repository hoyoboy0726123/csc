# 產品需求文件 (PRD)：AI 語音驅動全自動開發系統 (MVP)

## 1. 簡介 (Introduction)
本文件旨在定義「AI 語音驅動全自動開發系統」的最小可行產品（MVP）範圍。本系統是一個從語音構思到可執行程式碼的自動化流水線，旨在協助主管或開發者透過口述需求，快速產出結構化的產品需求文件（PRD）及完整的程式碼專案。

## 2. 專案願景與目標 (Vision and Goals)
*   **核心願景：** 實現「語音進，程式碼出 (Voice-in, Code-out)」的閉環開發流程。
*   **核心目標：**
    *   降低從想法到原型（Prototype）的轉化門檻。
    *   減少需求溝通落差，確保 PRD 與最終代碼的一致性。
    *   提供具備自我修復能力的開發助手，解決 AI 生成代碼的初期錯誤。

## 3. 目標族群 (Target Audience)
*   **專案主管/產品經理：** 需要快速驗證想法或建立原型。
*   **開發者：** 加速腳手架（Scaffolding）建立與基礎邏輯編寫。
*   **創業家：** 快速構建 MVP 以進行市場測試。

## 4. 功能需求 (Functional Requirements)

### 4.1 需求擷取與 PRD 轉化層
*   **多模態輸入：**
    *   支援即時錄音功能。
    *   支援上傳音檔（MP3/WAV 格式）。
    *   支援直接輸入文字需求。
*   **PRD 自動生成：**
    *   系統應自動節錄語音重點。
    *   生成 Markdown 格式之 PRD，包含：功能清單、業務流程描述、資料結構定義、API 規格草案。
*   **互動式修正機制：**
    *   使用者可針對生成的 PRD 輸入修改建議（如：「增加權限控管功能」）。
    *   AI 需根據建議進行滾動式更新，直到使用者滿意。

### 4.2 智慧代碼生成層
*   **框架自動決策：**
    *   AI 根據 PRD 內容自動選擇最適技術棧（如：Python 資料處理、React 前端、Node.js API）。
*   **全架構產出：**
    *   生成符合業界最佳實踐的資料夾結構。
    *   包含完整的業務邏輯與環境設定檔（如：`requirements.txt`, `package.json`, `Dockerfile` 等）。
*   **本地優先原則：**
    *   資料庫預設使用 SQLite 或本地檔案系統。
    *   若需第三方服務，產出詳細的配置說明文件。

### 4.3 雙模態代碼修復與優化
*   **手動錯誤訊息回傳：**
    *   使用者可將終端機執行的錯誤 Log 貼回系統，AI 需自動診斷並提供修正後的代碼。
*   **對話式功能微調：**
    *   使用者可透過口語描述修改細節（如：「將按鈕改為藍色」、「多增加一個日期欄位」），AI 進行增量修改。

### 4.4 系統設定與管理
*   **LLM 模型切換：**
    *   後台提供 **Gemini API Key** 與 **Groq API Key** 的設定與儲存。
    *   使用者可根據需求手動切換使用的 LLM 供應商（例如：長文處理用 Gemini，快速修復用 Groq）。

## 5. 使用者流程 (User Stories)
1.  **需求輸入：** 身為使用者，我可以上傳主管的錄音檔，讓系統自動產出初步的 PRD。
2.  **需求對齊：** 身為使用者，我可以在產出代碼前，透過對話不斷修正 PRD 內容，直到符合預期。
3.  **執行專案：** 身為使用者，我可以下載完整專案壓縮包，依照 README 說明在終端機安裝依賴後立即執行。
4.  **除錯與優化：** 身為使用者，若執行時報錯，我可以將錯誤訊息貼回系統，獲得修正後的版本。

## 6. 技術規格 (Technical Requirements)

### 6.1 開發平台與語言
*   **後端語言：** Python 3.9+ (推薦使用 FastAPI 或 Flask 處理 API)。
*   **異步處理：** 由於 LLM 響應時間較長，需支援非同步處理請求。

### 6.2 AI 與模型整合
*   **LLM APIs：**
    *   **Gemini API:** 用於長上下文 PRD 生成與複雜架構設計。
    *   **Groq API (Llama 3/Mixtral):** 用於即時對話修正與快速錯誤修復。
*   **語音轉文字 (STT)：** 整合 OpenAI Whisper 或 Google Speech-to-Text API。

### 6.3 資料儲存
*   **本地資料庫：** 使用 SQLite 儲存使用者的專案歷史、PRD 版本及代碼狀態，以支援多輪修復與上下文記憶。

### 6.4 代碼生成邏輯
*   **模板引擎：** 建立標準化的 Prompt 模板，確保產出的專案結構（Project Scaffold）一致。
*   **狀態管理：** 系統必須記錄當前代碼的最新版本，以便進行增量更新而非全盤重寫。

## 7. 交付物 (Deliverables)
1.  **結構化 PRD 文件：** Markdown 格式，支援預覽與下載。
2.  **原始碼壓縮包：** 包含完整目錄結構、邏輯代碼與環境配置。
3.  **執行說明文件 (README.md)：**
    *   專案簡介與功能說明。
    *   環境安裝指令（如 `pip install -r requirements.txt`）。
    *   啟動指令。
    *   第三方服務/API Key 配置指南。
4.  **設定面板：** 提供使用者配置 API Key 的網頁介面。

## 8. MVP 限制與範疇 (MVP Scope & Constraints)
*   **環境相依性：** MVP 階段不保證在所有環境下「一鍵成功」，但必須提供詳細的環境準備說明。
*   **安全性：** 代碼中嚴禁寫死（Hard-code）敏感資訊，應使用 `.env` 範例文件引導使用者配置。
*   **第三方服務：** 僅提供介面與對接邏輯，不包含第三方帳號的代申請。

## 9. 未來擴充建議 (Post-MVP)
*   **版本控制整合：** 直接整合 GitHub/GitLab API 推送代碼。
*   **雲端部署：** 支援一鍵部署至 Vercel 或 Docker 容器雲。
*   **自動化測試生成：** 在生成代碼的同時，自動產出單元測試（Unit Test）。
